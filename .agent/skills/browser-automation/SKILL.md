# ブラウザ自動実行ノウハウ (Browser Automation Know-how)

## 既知の問題と課題

### 1. 座標指定エラー (Viewport外へのアクセス)
* **現象**: `x coordinate must be less than viewport width` 等のエラー。
* **原因**: モバイル表示（例: 375x667）など、現在のビューポートサイズをエージェントが正確に認識せず、範囲外の座標をクリックしようとするため。
* **対策**: エージェントへの指示タスク内で「現在のビューポートサイズ（例: 375x667）を厳密に守ること」を明記する。

### 2. マウス操作・クリックのタイムアウト
* **現象**: `click_browser_pixel`, `browser_mouse_down`, `browser_mouse_up` などのアクションで `action timed out` が発生する。（Tetris画面のアクションボタンなどで確認済）
* **原因**: 
  - 絶対配置（`absolute`）された要素や、Reactの合成イベント（SyntheticEvent）に依存したコンポーネントに対して、自動ツールの低レイヤーのマウスイベントが正しく伝播・完了判定されない場合がある。
  - 要素が他のレイヤー（オーバーレイやCanvas）に隠れている。
* **対策**:
  - `click_browser_pixel` などの座標ベースの直接クリックを避け、CSSセレクタやテキストマッチを利用したDOMベースの操作を指示する。
  - テスト自動化の観点からは、ブラウザコンソールからJavaScriptを直接実行してイベントを発火させる（例: `document.querySelector('button').click()`）手法への切り替えを検討する。

### 3. テトリス画面における検証結果まとめ
* **できなかったこと**:
  - 推測した座標（X, Y）に対する直接のピクセルクリック（タイムアウト頻発）。
  - モバイルビューポート幅を超える座標に対する操作（即時エラー）。
* **今後の推奨アプローチ**:
  - 自動ツールのGUI操作に依存しすぎず、状態を操作する場合はWindowオブジェクト等にデバッグ用の関数を露出し、スクリプトから直接叩けるようにする。
  - 要素の検証にはDOMツリーの読み取り（`browser_get_dom`）と、確定したCSSセレクタによる操作をベースにする。
